考查点：preg_match 回溯绕过

看到题目名称，感觉有些奇怪，访问环境页面同样给出提示：true .swp file?

我们可以搜索一下 web swp文件

swp即swap文件，在编辑文件时产生的临时文件，它是隐藏文件，如果程序正常退出，临时文件自动删除，如果意外退出就会保留，文件名为 .filename.swp

由此推断该题应该是有一个以 .swp 为结尾的路径，我们使用 dirsearch 扫描一下

我们可以看到的确有一个这样的路径：/.index.php.swp

访问该路径得到一段源码

我们对源码进行分析，首先是自定义函数 jiuzhe

preg_match 函数是一个正则表达式匹配函数，在找到第一个匹配后即停止搜索，返回值只可能是 1（匹配成功）、0（未匹配） 或 false（出错）

我们可以通过正则表达式可视化工具看看

preg_match 匹配部分：

  • sys：匹配字符 “sys”
  • .*：匹配任意字符（除了换行符）0次或多次
  • nb：匹配字符 “nb”

当我们传入后首先会经过 if(!is_array($xdmtql))，也就是说我们输入的值不能是数组类型

is_array 函数用于检测变量是否为数组，如果是数组则返回 true，否则返回 false

接着会经过自定义函数 jiuzhe，按照当前逻辑，当我们的输入与 preg_match 部分不匹配时，才能进入 strpos 判断

strpos 函数用于查找一个字符串在另一个字符串中首次出现的位置（区分大小写），返回位置索引（从0开始），如果未找到则返回 false

此处需要我们的输入中有 sys nb 这样一个字符串，这与上面 preg_match 相悖

此时我们需要想想其他方法，也就是 preg_match 回溯绕过

在使用 preg_match 时，如果正则表达式的回溯次数超过了 PHP 的默认上限（通常为1000000次），preg_match 将返回 false，而不是 1 或 0

这样我们就可以绕过得到 flag

该字符串由 [、]、!、(、)、+ 6种字符组成，通过搜索得知这是一个名为JSFuck 的编码

JSFuck 是一种深奥的 JavaScript编程 风格。以这种风格写成的代码中仅使用 [、]、(、)、! 和 + 六种字符。此编程风格的名字派生自仅使用较少符号写代码的Brainfuck语言。

与其他深奥的编程语言不同，以JSFuck风格写出的代码不需要另外的编译器或解释器来执行，无论 浏览器 或 JavaScript引擎 中的原生 JavaScript 解释器皆可直接运行。

考查点：前端响应包、robots.txt、Cookie修改

访问题目链接，页面显示上图中信息，既然提到前端，我们不妨先看看源码

果不其然，在这儿得到提示：爬虫，由此我们很快就可以想到 robots.txt

Robots.txt 是站长或者建站系统创建的一个文件，目的是用来告诉蜘蛛怎么抓取自己网站的页面，主要通过 “allow” 和 “disallow” 指令来告知“user agents（用户代理）”

搜索引擎工作原理：

  1. 爬行：持续性的收集网页，建立数据库
  2. 索引：分析网页主题，根据主题建立索引
  3. 排名：去索引库匹配页面，排序后展示

在爬取互联网上的网页的时候，爬虫会先查看网站是否有 robots.txt 文件，如果有的话，它会先阅读文件上的规则，然后根据列出的规则去爬行网站

访问 /robots.txt，我们得到了 flag3:c0ad71dadd11}

然后就没有其他提示了，此时我们可以抓包看看，看看响应包中会不会隐藏有其他部分的 flag

好家伙，在响应中我们果然有找到 Flag2: 3ca8737a70f029d

F12 –> 网络 –> 刷新页面，查看响应包

这种方式同样也能得到flag2

现在我们就差第一部分flag了，然而已经没有其他提示信息了，但我们忘了还有一个重要的地方，那就是 Cookie

F12 --> 应用 --> Cookie，在这里我们发现有一个名为 id 的 Cookie，其值为 0

猜测 id 取不同值时，页面显示可能会不同（可以将这想象成一个是普通用户user，一个是管理员用户root）,所以尝试将 id 值改为 1，然后刷新页面（此处需要注意：刷新页面后，id 的值会还原为 0，但此题不影响后续做题），果然得到的不同的回显：flag1:flag{7ac5b

最后将三段 flag 合成即为最终 flag

考查点：JWT伪造

访问题目链接，我们看到有登录、注册两个功能

先尝试一下弱口令、sql万能密码，看看能不能直接登录

从结果来看显然是不行的，但我们做题时还是要多多尝试

既然无法直接用登录，那么我们就注册一个用户，先尝试看能不能注册一个 admin

尝试后提示“该用户名或邮箱已被注册！”，从这我们知道大概率是存在 admin 导致的

所以我们只能另外注册其他用户

注册完成后进行登录，成功进入系统

查看系统中其他信息，并未发现有flag的任何提示，此时，我们需要看看题目 JWT 会不会是一种提示信息

JWT 是 JSON Web Token 的缩写，广泛应用于用户身份验证和授权，尤其在跨域认证场景中非常流行

JWT 本质上就是一组字串，通过（.）切分成三个为 Base64 编码的部分：

  • Header（头部） : 描述 JWT 的元数据，定义了生成签名的算法以及 Token 的类型。Header 被 Base64Url 编码后成为 JWT 的第一部分。
  • Payload（载荷） : 用来存放实际需要传递的数据，包含声明（Claims），如sub（subject，主题）、jti（JWT ID）。Payload 被 Base64Url 编码后成为 JWT 的第二部分。
  • Signature（签名）：服务器通过 Payload、Header 和一个密钥(Secret)使用 Header 里面指定的签名算法（默认是 HMAC SHA256）生成。生成的签名会成为 JWT 的第三部分。

通过搜索我们得知 JWT 是用于确认用户身份的，查看 Cookie，我们可以看到当前用户下 JWT 的值

使用 jwt.io 解析，发现缺少密钥

此时我们需要用到一个工具 c-jwt-cracker 爆破密钥

得到密钥后，我们就可以改写令牌，将用户名改为 admin，重新生成 JWT

将改好的 JWT 值替换原来 Cookie 中 JWT 的值，刷新页面，此时我们就以 admin 的身份成功登录了

查看个人中心即可得到 flag
